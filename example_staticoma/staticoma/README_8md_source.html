19 Terminology
20 -----------
21 
22 The following terms are not common, but useful for definition of the problems
23 addressed by staticoma:
24 
25 - Configuration parameters can be divided into three categories:
26  1. build-time -- values are known during compilation and never change;
27  2. launch-time -- values are known at launch time and never change during
28  execution;
29  3. run-time -- those that change dynamically during execution.
30 
31 - Stack -- a robot-specific set of packages providing multiple services and
32  configuration parameters for them. Usually there are at least two partially
33  overlapping stacks for a specific robot: one for simulation, and one for
34  deployments.
35 
36 - Configuration composition -- construction of configuration parameters from
37  multiple, potentially overlapping pieces scattered among multiple (ROS)
38  packages in a stack. Roughly speaking default parameters defined by packages
39  providing certain functionality are overlayed with robot-sepecific
40  parameters.

44 Standard ROS approach
45 ---------------------
46 
47 `roslaunch` (http://wiki.ros.org/roslaunch) -- ROS subsystem designed for
48 ROS-specific service management, provides two options to pass parameters to a
49 specific node (service):
50 
51 - individual parameters can be passed via command line, but due to nested
52  structure of roslaunch scripts such parameters often must be passed from top
53  level to lower level scripts through a chain of intermediate scripts;
54 
55 - parameters can be uploaded from YAML files to 'parameter server'
56  (http://wiki.ros.org/Parameter%20Server) -- ROS service which manages global
57  parameters at run-time, sequential uploading of parameters allows
58  configuration composition.
59 
60 `roslaunch` ensures that parameters declared in the stack launch script tree
61 are uploaded to parameter server in the order of their appearance before any of
62 the nodes are started.

66 Scope and goals of `staticoma`
67 ------------------------------
68 
69 `staticoma` addresses build-time and launch-time parameter management in order
70 to alleviate some of parameter server issues:
71 
72 1. ROS allows recording data exchange between nodes using `rosbag` utility. The
73  problem is that the content of ROS parameter server is not saved to bag
74  files, but often affects data exchange. `staticoma` introduces a workaround
75  that allows storing configurations in bag files, making them
76  self-sufficient.
77 
78 2. Launch-time configuration composition performed by `roslaunch` may be
79  confusing since configuration pieces are stored in different locations and
80  their precedence may not be straightforward. This problem is alleviated by
81  performing build-time parameter composition to a single configuration file.
82 
83 3. Node parameters may be removed or renamed during development, it would be
84  nice to have a mechanism to enforce parameter consistency between 'parameter
85  consumer' and 'parameter provider' packages. This goal is achieved by
86  preventing providers from adding new parameters during build-time
87  composition.
88 
89 4. ROS parameter server has been deprecated in ROS2, and it is a good idea to
90  develop services that do not rely on it.
91 
92 5. `roslaunch` is inferior in many aspects to generic service managers such as
93  `systemd`, but replacing it is very difficult due to deep integration with
94  parameter server. Limited dependency on parameter server would allow to
95  explore alternatives to `roslaunch`.

99 Key design concepts
100 -------------------
101 
102 ### Handling build-time parameters
103 
104 1. All build-time parameters of a stack are written to a single file during
105  compilation ('stack-config') using custom `cmake` functions.
106 
107 2. stack-config is composed step by step by overlaying parameter files provided
108  by package dependencies. For example: package B depends on A, A defines
109  build-time parameters
110  (https://github.com/asherikov/staticoma_workspace/blob/master/src/package_a/CMakeLists.txt#L15):
111 ```
112 staticoma_export("config/base_config.yaml")
113 ```
114 and B builds on top of them
115 (https://github.com/asherikov/staticoma_workspace/blob/master/src/package_b/CMakeLists.txt#L17)
116 ```
117 staticoma_compose(
118  merged_config.yaml
119  package_a base_config.yaml
120  package_b "config/extended_config.yaml"
121 )
122 ```
123 `merged_config.yaml` can in turn be used by other packages
124 (https://github.com/asherikov/staticoma_workspace/blob/master/src/package_c/CMakeLists.txt#L28),
125 etc.
126 
127 3. If it is necessary to prevent package B from addiing new parameters to
128  configuration file provided by A, then `STRICT` keyword can be used, e.g.,
129 ```
130 staticoma_compose(
131  merged_config.yaml
132  STRICT
133  package_a base_config.yaml
134  package_b "config/extended_config.yaml"
135 )
136 ```
137 
138 4. There exist different merging strategies for `YAML` arrays, `roslaunch`
139  supports only 'overwrite' strategy, where the contents of an array is
140  completely overriden if the redefined in overlaying parameters. `staticoma`
141  uses the same strategy by default, but also supports 'append' strategy,
142  which extends arrays with new members, e.g.
143  (https://github.com/asherikov/staticoma_workspace/blob/master/src/package_c/CMakeLists.txt#L20):
144 ```
145 staticoma_compose(
146  merged_config.yaml
147  ARRAY_MERGE_STRATEGY append
148  package_a base_config.yaml
149  package_b extended_config.yaml
150 )
151 ```
152 
153 5. Global or partial (in some cases) stack-config can be used for partial stack
154  launches, which is convenient for development.
155 
156 6. A dedicated service `staticoma/server` reads stack-config on launch and
157  advertises it as a static topic (`stack-config-topic`), which can be
158  recorded in a bag file.
159 
160 7. It is not very safe to rely completely on `staticoma` server, therefore
161  nodes should prefer reading stack-config directly from a file.
162 
163 8. Passing of stack-config filename:
164  - environment variable -- should be used by default
165  (https://github.com/asherikov/staticoma/blob/master/test/client.test#L4);
166  - command line argument -- can be provided by the user if necessary.
167 
168 9. Initialization of stack-config in a node:
169  1. if stack-config filename is given explicitly, open this file, fail if
170  does not exist.
171  2. if environment variable specifying stack-config filename exists, use
172  it to locate the file, fail if does not exist.
173  3. if stack-config filename cannot be determined try accessing
174  `stack-config-topic`, fail if not available.
175  4. Once the content of stack-config has been received, the node can parse
176  and use it as necessary.
177 
178 10. Some parameters, such as robot description, must be set in parameter
179  server, so we cannot get rid of parameter server completely. `staticoma`
180  server can dump all parameters and publish them in a message to be saved in
181  bag files.
182 
183 11. Restoring of parameter server content from a bag file should be performed
184  by an additional `replayer` service that would be executed alongside
185  `rosbag play`.
186 
187 12. `replayer` can be started in two ways:
188  - as a normal ROS node --
189  https://github.com/asherikov/staticoma/blob/master/test/client_replay1.test#L7),
190  in this case parameters are not guaranteed to be uploaded before other
191  nodes are started;
192  - inside `<param>` roslaunch tag --
193  https://github.com/asherikov/staticoma/blob/master/test/client_replay3.test#L7.
194

196 ### Handling launch-time and run-time parameters
197 
198 1. Although launch-time parameter values are unknown at build-time, we usually
199  know which services provide them, in case of roslaunch scripts this
200  information is implicitly hardcoded.
201 
202 2. A unique topic can be allocated for each dynamic provider known at build
203  time. Publishing parameters on topics is also going to make them
204  'rosbaggable'. Topic name collisions can be avoided by using naming
205  conventions, e.g.,: `/<consumer_id>/<provider_id>/<topic_name>`.
206 
207 3. Parameter consumer may receive parameters from multiple sources, conflict
208  resolution is implementation specific, for example, overlay them in order of
209  provider declaration, or instantiate a separate processing thread for each.
210 
211 4. roslaunch `arg`, `param`, and `rosparam` tags can be replaced with nodes
212  starting rostopic with `rostopic pub --latch` nodes, e.g.
213  https://github.com/asherikov/staticoma_workspace/blob/master/src/package_b/launch/node.launch#L3.
214 
215 5. Consuming parameters
216  (https://github.com/asherikov/staticoma_workspace/blob/master/src/package_c/src/node.cpp#L101):
217  1. obtain stack-config to identify parameter providers.
218  2. wait for published parameters
219  3. parse parameters
220 
221 
222 ### Parameter representation
223 
224 1. `staticoma` does not depend on parameter representation, it can be `JSON`,
225  `YAML`, `XML`, etc, as long as we have tools to compose them at build-time
226  and parse in nodes. Currently, only `YAML` is supported.
227 
228 2. Configuration files are passed raw (plain YAML, etc): performance and
229  throughput are not important since parameters are static in run-time and
230  need not be frequently transferred.
231 
232 
233 ### Other considerations
234 
235 1. If we remove all conditional logic, arguments, and parameters from launch
236  files, they are pretty much build-time configuration files, which can be
237  composed at build-time as well. This may be useful if we would like to use
238  something like `supervisord` for service management, since it uses single
239  configuration file instead of multiple 'unit'/'launch' files.
240 
241 2. It could be interesting to consider migration from YAML to jsonnet. Jsonnet
242  is a data templating language (https://jsonnet.org/) which has build-in JSON
243  merging functionality and a lot of other functions.
